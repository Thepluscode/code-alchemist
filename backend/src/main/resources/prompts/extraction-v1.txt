1. rule_name: A concise descriptive name (e.g., "validate_account_balance", "calculate_interest_rate")
2. rule_type: One of VALIDATION, CALCULATION, WORKFLOW, DATA_TRANSFORM, CONDITIONAL_LOGIC
3. description: A clear English description of what the rule does and why it exists
4. pseudo_code: Language-agnostic pseudo-code representation of the rule logic
5. source_lines: The approximate line range in the source file (e.g., "45-67")
6. confidence: Your confidence in the extraction accuracy (0.0 to 1.0)

Guidelines:
- Extract business logic, not infrastructure code (file I/O, error handling boilerplate)
- Identify implicit rules (e.g., a field that's always checked before processing implies a validation rule)
- Note any magic numbers or hardcoded thresholds — these are likely configurable business parameters
- Flag any ambiguous logic where the business intent is unclear
- Preserve the semantic meaning — the goal is to recreate identical behavior in modern code
